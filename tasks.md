# PCCA — Refactoring Backlog

This is the authoritative list of known gaps between the implementation and [architecture.md](./architecture.md) / [scenarios.md](./scenarios.md). Each task has a priority, a concrete problem, a sanctioned solution, acceptance criteria, and the impact of fixing it.

**Rules:**
- P0 tasks block daily-use dogfood. Do not start P1+ work while P0 items are open.
- New features land only after the related P0 tasks are closed.
- Every non-trivial PR should reference a task ID (T-01 … T-27). If no task exists, add one here first.

**Priority semantics:**
- **P0** — critical correctness / reliability; user-visible silent failure today.
- **P1** — required for 14-day-use stability; spec-mandated reliability controls.
- **P2** — quality of outcomes; makes the curation actually learn / explain.
- **P3** — polish, consistency, observability, nice-to-haves.

| T-01 | P0 | Session challenges are silent. Telegram service announces needs-reauth sources on `/setup` and after failed runs. | (1) Simulated login-wall page yields `follow_state='needs_reauth'` and pipeline continues. (2) `/setup` in Telegram lists sources currently needing re-auth. (3) A fresh login flips state back to `active`. | Eliminates the #1 silent-failure mode; users see reality instead of empty "success" digests. |
| T-02 | P0 | Feedback buttons have no `item_id`. `fb:up|down|save` callbacks resolve only the subject; `feedback_events.item_id` is always NULL. The "negative feedback reduces similar items within 3 cycles" guarantee in [scenarios.md §Scenario 5](./scenarios.md#acceptance-criteria-4) is mechanically impossible. | Encode a compact reference in `callback_data` — either `fb:<action>:<digest_item_pk>` (Telegram limit = 64 bytes) or a monotonic short token mapped in a `digest_buttons` table. Handler looks up `digest_items → item_id`. Add `UNIQUE(subject_id, item_id, feedback_type)` on `feedback_events` for dedup. | (1) Clicking 👎 on digest item N records `feedback_events(subject_id=..., item_id=..., feedback_type='button_down')`. (2) Second click on the same button is a no-op. (3) A unit test asserts the mapping round-trip. | Unblocks the learning loop. Without this, T-17 cannot deliver. |
| T-03 | P0 | Pass-2 is not gated on a shortlist. `orchestrator.py` runs Pass-1 score + Ollama rerank on every item inline, per subject × source × item. With ~200 items / night × 25s httpx timeout, worst case is ~80 min wall time. Spec ([architecture.md §6.2](./architecture.md#62-pass-2--deep-curation), [scenarios.md §Scenario 3 step 8-9](./scenarios.md#pipeline-steps)) requires shortlist. | Restructure orchestrator to: (a) collect all items globally, (b) per subject Pass-1 score all items, (c) take top-K (default K=20) per subject, (d) Pass-2 / model rerank on top-K only, (e) persist scores, (f) digest picks top 3–5. | T-04 | P0 | Scheduler fire-and-forget is unsafe. `scheduler.py::_run_async_job` creates a task from a sync APScheduler callback without retaining the reference, without awaiting, without handling exceptions. (2) No `RuntimeWarning: coroutine was never awaited` in logs. (3) Restarting the app does not leak tasks. | Eliminates the "scheduled run silently didn't happen" failure class. |

(2) The second call is observably a no-op or a resend, not a new composition. (3) Schema migration v3 applies cleanly on a fresh DB and an existing DB. | Eliminates duplicate-digest spam and makes the delivery layer actually idempotent as the spec promises. |
| T-06 | P0 | `sources.last_crawled_at` is never read or written. Item upsert should not clobber `raw_text` unless content actually changed. | (1) Second nightly run on the same inputs writes 0 new items and 0 updated items. (2) An item recorded while logged-in does not get overwritten by a logged-out empty body. (3) `last_crawled_at` is visible in `pcca list-sources` output or an equivalent diagnostic. | Prevents historical data corruption and eliminates wasted rescoring work. |
| T-07 | P1 | No crawl throttling or jitter. Every collector uses fixed `page.wait_for_timeout(N)`; no per-platform rate limits; no randomized delays. (2) Breaching the per-platform budget queues the next call rather than firing immediately. (3) Configurable via env vars. | Reduces ban risk on X / LinkedIn and matches spec §8.1. |
| T-08 | P1 | No circuit breaker on failing sources.
